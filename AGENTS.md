# Project Notes For Codex

## Question Handling Rules

- Если пользователь задаёт вопрос в запросе, сначала отвечать на вопрос текстом и не редактировать код или файлы автоматически.
- Это правило действует даже если вопрос выглядит риторическим или подразумевает возможную правку; к изменениям переходить только после явной просьбы сделать, поправить или реализовать.

## Local Search Rules

- На этой машине `rg`/`ripgrep` в проекте может падать с `Access is denied`.
- Текстовые файлы проекта обычно сохранены как UTF-8 без BOM, а Windows PowerShell 5.1 без явного параметра читает их как ANSI/1251 и ломает русскоязычный текст в mojibake.
- Для чтения текстовых файлов всегда указывать `-Encoding UTF8`: `Get-Content -LiteralPath path\to\file.md -Encoding UTF8`.
- Для поиска по содержимому всегда указывать `-Encoding UTF8` у `Select-String`: `Get-ChildItem -Recurse -File | ... | Select-String -Pattern '...' -Encoding UTF8`.
- Если вывод снова выглядит как `РќР°...` или `Р”Рѕ...`, считать чтение некорректным и повторить команду с явным `-Encoding UTF8`, не делать выводы по битому тексту.
- Для поиска по файлам сразу использовать обычный PowerShell-поиск, но только после фильтрации путей и типов файлов: `Get-ChildItem -Recurse -File | Where-Object { ...исключения... } | Select-String -Pattern '...' -Encoding UTF8`.
- Для поиска конкретных файлов использовать `Get-ChildItem -Recurse -Filter ...` или `Get-ChildItem -Recurse -File | Where-Object { ... }`.
- Никогда не читать и не искать без явной необходимости в тяжелых/сгенерированных каталогах: `node_modules`, `__pycache__`, `.git`, `.gradle`, `.gradle-user-home`, `.kotlin`, `.venv`, `venv`, `dist`, `build`, `.next`, `.cache`, `coverage`, `tmp`, `temp`.
- При рекурсивном PowerShell-поиске всегда исключать такие каталоги ДО `Select-String`, чтобы не забивать контекст и не тратить токены на зависимости, кеши, сборочные артефакты и бинарные файлы.
- Широкий поиск по содержимому должен ограничиваться исходниками, ресурсами и документацией. По умолчанию искать только в текстовых расширениях: `.kt`, `.java`, `.xml`, `.gradle`, `.kts`, `.md`, `.txt`, `.yaml`, `.yml`, `.json`, `.properties`, `.toml`, `.go`, `.c`, `.cpp`, `.h`, `.hpp`, `.sh`, `.ps1`.
- Никогда не запускать широкий поиск по базам данных, бинарникам и артефактам: `.db`, `.sqlite`, `.sqlite3`, `.realm`, `.bin`, `.jar`, `.aar`, `.apk`, `.aab`, `.dex`, `.so`, `.dll`, `.exe`, `.class`, `.zip`, `.7z`, `.gz`, `.tar`, `.png`, `.jpg`, `.jpeg`, `.webp`, `.gif`, `.pdf`.
- Если после широкого поиска стало видно, что команда зацепила `.gradle`, `build`, кэш, базу данных или бинарный файл, считать поиск некорректным: не использовать его вывод для выводов по задаче и повторить поиск с более строгим `Where-Object` по директориям и расширениям.
- Если для задачи все-таки нужен файл внутри такого каталога, читать только конкретный известный файл или короткий точечный фрагмент, а не запускать широкий поиск по всему каталогу.

## Logging Rules

- Все новые и изменяемые логи писать на русском языке.
- Каждый лог должен начинаться с контекста функции или этапа в формате `"[functionName]"`.
- Пример: `Log.i(TAG, "[setStateInternal] Поднимаем туннель " + tunnel.getName() + ' ' + state);`
- Логи должны помогать разбирать поведение кода и отладку, а не существовать для галочки.
- Избегать бессмысленного спама в логах, но не избегать полезных логов.

## Roadmap Rules

- Периодически поглядывать в `roadmap.md`, особенно перед реализацией новых фич, инфраструктурных задач и уборки технического долга.
- Если в текущей задаче была прямо сейчас реализована фича или техзадача, уже упомянутая в `roadmap.md`, обновить соответствующий пункт в `roadmap.md` отметкой `реализовано YYYY-MM-DD`.
- Не начинать выполнять задачи из `roadmap.md` только из-за их наличия в файле: это список отложенных задач, а не автоматический план на текущую сессию.

## Code Rules

- Форматируй строки через f"чтото {variable}" форму всегда если только не оправдано иное типо надо вызывать функцию чтоб получить значение для строки, вызов которой цепляет еще какое то поведение
- Никогда не запускать Gradle/gradlew/gradlew.bat/Android Studio build-команды. Gradle в этом проекте запускает только пользователь.

## Documentation Comments

- Для каждого создаваемого или редактируемого класса добавлять или обновлять doc-комментарий.
- Для каждой создаваемой или редактируемой функции добавлять или обновлять doc-комментарий.
- Doc-комментарии должны кратко объяснять назначение сущности и важные особенности поведения.

## Internal Code Version

- После каждого изменения кода, сделанного Codex, инкрементить внутреннюю версию в строковом ресурсе `rnk_internal_code_version`.
- Ресурс хранится в `ui/src/main/res/values/strings.xml` и отображается в нижней строке лицензии на главном RNK-экране.
- Формат значения: `\u0020в-N`, где `N` — возрастающий внутренний номер, не связанный с Gradle `versionCode` или `versionName`.

## Module Documentation

- В проекте может быть список doc-файлов по отдельным сложным местам проекта. Если работа затрагивает такое место и соответствующий файл существует, сначала обратиться к нему.
- Если изменение кода меняет актуальность такого doc-файла, обновить этот doc-файл в той же задаче.
- Doc-файлы по сложным местам проекта хранить в папке `docs/`.
- Каждый такой doc-файл должен быть умеренно наглядным для человека и удобным для ИИ: писать кратко, однозначно, структурно и без лишней воды.
- Там, где это помогает понять или проверить описание, doc-файл должен ссылаться на реальные файлы проекта, функции и строки кода.

## Module Documentation Files

- `docs/vk-call-source-flow.md` — сценарии работы с VK-токеном, ссылкой на звонок, backend VK-данными и запуском TURN-туннеля.
- `docs/vk-turn-config-contract.md` — точный формат `.conf` TURN-комментариев и JSON-ответа `/api/v1/configs/request`, который приложение умеет подхватывать.
- `docs/turn-stream-liveness.md` — поддержание жизни TURN stream-ов, DTLS watchdog, WRAP cover packets и почему они не заменяют heartbeat.
- `docs/vk-turn-go-captcha-flow.md` — Go/native контракт runtime-капчи: как stream отдаёт `redirect_uri` наружу и ждёт `success_token`.
- `docs/vk-turn-tunnel-runtime.md` — чтение WireGuard/AmneziaWG-конфигов, хранение TURN-настроек, runtime-подмена endpoint и полный цикл запуска VK/TURN-туннеля.
- `docs/vk-turn-wrap-deployment-notes.md` — практические заметки по внедрению WRAP, сборке Android Go-бинарника, запуску серверного `vk-turn-proxy`, портовому редиректу и диагностике.


Legend Documentation

- В проекте может быть отдельный набор legend-doc файлов: это документы про внешние знания, протоколы, API, сервисы, DEX-ы, блокчейн-механику и рыночные особенности, связанные с проектом.
- Legend-doc отличается от module-doc тем, что описывает не внутреннее устройство нашего кода, а внешний мир, с которым проект взаимодействует.
- Legend-doc файлы хранить в папке `docs/legend/`.
- Если задача затрагивает внешний сервис, протокол или API, для которого уже есть legend-doc файл, сначала обратиться к нему.
- Если в ходе работы обнаружена новая важная информация о внешнем сервисе, API, формате данных, ограничениях, рисках или надежном способе проверки, обновить соответствующий legend-doc файл в той же задаче.
- Если подходящего legend-doc файла нет, но знание явно повторно пригодится проекту, создать новый файл в `docs/legend/`.
- Legend-doc должен быть практичным: фиксировать проверенные наблюдения, рабочие способы взаимодействия, известные ограничения, риски, примеры запросов/ответов и ссылки на источники, если они есть.
- Непроверенные гипотезы можно записывать только в отдельном разделе `Гипотезы`, явно помечая их как непроверенные.
- Устаревшие сведения не удалять молча: переносить в раздел `Устарело` или заменять с кратким пояснением, почему информация больше неактуальна.
